webpack 5 简介

Posted on 2020-12-12 Edited on 2020-12-15

webpack 5 简介

关于webpack v5.0.0

主要改变

  • 通过持久缓存(Persistent Caching)提高构建性能
  • 用更好的算法和默认值来改进长期缓存(Long Term Caching)
  • 通过更好的 Tree Shaking 和 Code Generation 改善 bundle size
  • 改善与 Web 平台的兼容性
  • 在不引入 breaking change 的情况下，清理那些在实现 v4 时处于奇怪状态的内部结构
  • 通过引入 breaking change 为未来的功能做准备，并允许我们尽量长时间保持在v5

关于持久缓存（Persistent Caching）

什么是持久缓存？

https://jackrabbit.apache.org/oak/docs/nodestore/persistent-cache.html

文件的持久性缓存旨在用于文档和对象的中期存储。

文缓存可以选择性的使用持久缓存。在早期版本中，这种缓存类似于内存中缓存，但是除了将最近使用的节点保留在内存中之外，它还将它们存储到磁盘中。这样，很多读取行为将从存储后端（例如MongoDB）读取替换为本地磁盘读取。如果从本地磁盘读取的速度比从存储后端读取的速度快，这将特别有用。除此之外，持久缓存还可以减轻存储后端的负载。

Webpack 怎么做持久缓存？

https://webpack.js.org/blog/2020-10-10-webpack-5-release/#persistent-caching

https://github.com/webpack/changelog-v5/blob/master/guides/persistent-caching.md

#####一个可选的文件系统缓存

缓存在哪？

缓存将默认存储在 node_modules/.cache/webpack（当使用 node_modules 时）或 .yarn/.cache/webpack（当使用 Yarn PnP 时）中。当所有的插件都正确处理缓存时，你可能永远都不需要手动删除它。

谁会使用？

许多内部插件也会使用持久性缓存。例如 SourceMapDevToolPlugin (缓存 SourceMap 的生成)或 ProgressPlugin (缓存模块数量)

#####什么策略？

https://webpack.js.org/configuration/other-options/#cache

持久性缓存将根据使用情况自动创建多个缓存文件，以优化对缓存的读写访问。

算法使用hash算法，默认为md4

默认情况下，时间戳将用于开发模式的快照，而文件哈希将用于生产模式。文件哈希也允许在 CI 中使用持久性缓存。

什么影响？——监视模式下的增量构建速度很快，但是重新启动Webpack可能会非常慢。
关于编译器的空闲和关闭

现在编译器需要在使用后被关闭。编译器会进入/离开空闲状态，并且有这些状态的钩子。一些插件可能是用这些钩子来做不重要的工作（例如持续缓存会缓慢的存储缓存到磁盘上）。（默认在空闲10000ms时，会开始进行缓存存储cache.idleTimeout）。在编译器关闭时——所有剩余的工作应该尽快完成。将会有一个callback标志关闭完成。

插件及其作者应该注意，某些用户会忘记关闭编译器。所以，所有的工作在空闲状态也可以下最终完成。当工作正在进行时，我们应该放置进程退出。

当被传递回调时，webpack()方法将会自动调用close

迁移：将使用Node.js API时，要保证完成工作时调用Compiler.close。

######关于文件生成

过去：

总是在第一次构建时发出所有的输出文件，但在增量（观察）构建时跳过了写入未更改的文件。（假设在 webpack 运行时，没有任何其他东西改变输出文件。）

现在：

增加了持久性缓存后，不能认为即使在 webpack 不运行时也没有其他东西改变输出目录。

所以 webpack 现在会检查输出目录中现有的文件，并将其内容与内存中的输出文件进行比较。只有当文件被改变时，它才会写入文件。 这只在第一次构建时进行。在运行中的 webpack 进程中，任何增量构建都会在生成新的资产(assets)时写入文件。

我们怎么用？

https://github.com/webpack/changelog-v5/blob/master/guides/persistent-caching.md

首先，请注意默认情况下不启用持久性缓存。我们必须选择使用它。

这是为什么？webpack试图强调安全性而非性能。我们不想默认启用一项功能，该功能将使您的性能提高95％，但会使您的应用程序/工作流/构建中断5％。

听起来好像坏了，但相信我，事实并非如此。但这需要开发人员采取额外的步骤来正确配置它。

我们可以直接使用序列化和反序列化，但是无法立即使用缓存失效

缓存失效：webpack需要找出什么时候缓存不再有效，并停止将其用于构建，

因此，当在应用程序中更改文件时，就会发生缓存失效

例如：更改了magic.js， webpack必须使magic.js的缓存条目失效。构建将会再次处理这个文件。例如 运行babel、typescript等等，然后解析文件并再次运行code generation。然后，webpack可能还会使bundle.js的缓存条目无效，并且构建将从包含的模块中重建该文件。

为此webpack会跟踪每个模块的fileDependencies contextDependencies和missingDependencies，并创建文件系统快照。将该快照与真实文件系统进行比较，并且当检测到差异时，将触发该模块的重建。

然后，对于的缓存条目bundle.js，webpack将存储一个etag，这是所有贡献者的哈希值。比较etag，并且仅当它与缓存条目匹配时才可以使用

以上这些对于webpack 4中的内存缓存也是必要的。并且以上特性开发人员可以直接使用，而无需额外的配置。对于webpack 5中的持久缓存，存在新的挑战。

webpack还需要使缓存条目无效：

  • 当您npm升级加载程序或插件时
  • 当您更改配置时
  • 当您更改正在配置中读取的文件时
  • npm升级配置中使用的依赖项时
  • 当您将不同的命令行参数传递给构建脚本时
  • 当您拥有自定义的构建脚本并进行更改时

关于长期缓存

webpack 怎么做长期缓存？

修改了某个模块的代码，不会破坏其他模块的缓存，这就是我们想要实现的持久性缓存

确定的 Chunk、模块 ID 和导出名称

新增了长期缓存的算法。这些算法在生产模式下是默认启用的。

在以前，默认情况下，Webpack根据顺序为模块分配整数ID。因此，在更改模块时，所有ID可能都会更改，从而使缓存无效。

该算法以确定性方式为模块和块分配短（3或5位数字）数字ID，并为导出分配短（2个字符）名称。这是包大小和长期缓存之间的折衷方案。由于这些配置将使用确定的 ID 和名称，这意味着生成的缓存失效不再更频繁。

真正的内容哈希

当使用[contenthash]时，Webpack 5 将使用真正的文件内容哈希值。之前它 “只 “使用内部结构的哈希值。当只有注释被修改或变量被重命名时，这对长期缓存会有积极影响。这些变化在压缩后是不可见的。

关于改善 bundle size

tree-shaking

import()允许通过/* webpackExports: ["abc", "default"] */魔术注释手动摇动模块。

嵌套

webpack 现在能够跟踪对导出的嵌套属性的访问。（清除未使用的导出和混淆导出）

在这个例子中，可以在生产模式下删除导出的b。

内部模块

webpack 4 没有分析模块的 export 和 import 之间的依赖关系。webpack 5 有一个新的选项 optimization.innerGraph，在生产模式下是默认启用的，它可以对模块中的标志进行分析，找出 export 和 import 之间的依赖关系。

内部依赖图算法会找出 something 只有在使用 test 导出时才会使用。这允许将更多的 export 标记为未使用，并从代码包中省略更多的代码。

注意：使用 eval() 将为一个模块放弃这个优化，因为经过 eval 的代码可以引用范围内的任何标记。这种优化也被称为深度范围分析。

CommonJs

webpack 曾经不进行对 CommonJs 导出和 require() 调用时的导出使用分析。

webpack 5 增加了对一些 CommonJs 构造的支持，允许消除未使用的 CommonJs 导出，并从 require() 调用中跟踪引用的导出名称。

code generation

webpack会检测何时发生ASI，并且在未插入分号时会生成较短的代码。Object(...)->(0, ...)

webpack将多个导出 getter 合并为一个运行时函数调用：r.d(x, "a", () => a); r.d(x, "b", () => b);->r.d(x, {a: () => a, b: () => b});

webpack 4仅用于发出ES5代码。webpack 5现在可以生成ES5和ES6 / ES2015代码。

仅支持现代的浏览器，将使用箭头功能生成较短的代码，并使用constTDZ for的声明生成更多符合规范的代码export default。

关于改善与 Web 平台的兼容性

不再为 Node.js 模块 自动引用 Polyfills

早期，webpack的目标是允许在浏览器中运行大多数Node.js模块，但是模块格局发生了变化，现在许多模块用途的编写主要是出于前端目的。webpack <= 4带有许多Node.js核心模块的polyfill，一旦模块使用了任何核心模块（即crypto模块），这些模块就会自动应用。

虽然这使使用为Node.js编写的模块变得容易，但它会将这些巨大的polyfill添加到包中。在许多情况下，这些填充材料是不必要的。

webpack 5会自动停止填充这些核心模块，并专注于与前端兼容的模块。我们的目标是提高与Web平台的兼容性，在Web平台上，Node.js核心模块不可用。

如果你的代码中有引用这些 Node.js 的模块，要升级到 webpack 5, 将尽量使用前端的模块，或者自行手动添加适合的 Polyfills。

而针对那些类库的开发者，请在 package.json 中定义 browser 字段，使类库在前端能适用。

基于angular的自动化测试探索

基于angular的自动化测试探索

框架选择—— Jest vs Karma

https://charith-rhettiarachchi.medium.com/why-use-jest-over-karma-for-angular-testing-b56ffa82f8

JSET建立在Jasmine之上。

Jasmine是BDD的测试开发框架。零依赖且不需要DOM

Jest是基于Jasmine构建的，拥有Jasmine所有开箱即用的功能，以及KARMA的功能。另外由于Jest是Facebook创建的，因此Facebook所有的JavaScipt库和框架，包括React。

Karma是angular团队构建的测试运行程序，在自己的浏览器环境中运行，也就是打开浏览器以运行测试。

  • Jest比Karma测试快2到3倍
  • Jest拥有快照测试
    • 首次运行测试时，可以将.toMatchSnapshot（）添加到组件中。
  • 更好的持续集成
    • 运行JsDOM测试，测试环境中不需要安装浏览器（由于Jest不在真正的浏览器上运行（使用jsdom）
    • 因此存在jsdom与目标浏览器不同的潜在风险*
  • 更少的配置
    • Jest的哲学之一是提供集成的“零配置”体验

关键概念

TestBed

这是 angular测试模块的基本构件块，它实际上创建了一个模拟NgModule的动态构建的测试模块，输入TestBed.configureTestingModule()的元数据也跟NgModule中的很相似。

实际上TestBed创建了一个动态测试模块，而beforeEach() 则是在每个case运行前调用的。

compileComponents()

如果要使用外部文件创建组件，例如styleUrls、templateUrl，则需要在创建组件之前从文件系统读取这些文件，这是异步操作，因此整个配置都放在异步函数中。

实际上，compileComponents是异步的从文件系统读取外部文件并编译相应的组件，如果我们使用angular cli运行测试则无需执行此操作，因为cli在测试前会编译所有的代码。

为什么需要有两个beforeEach方法？

上文提到，compileComponents 是异步函数，所以我们希望在创建用于访问组件实例的装置（fixture）前，确保组件已经编译成功。

NO_ERRORS_SCHEMA

在测试app.component时忽略所有无法识别的标签，例如page.component中的标签。如果不使用NO_ERRORS_SCHEMA ，则需要在declarations中声明所有用到的组件。

最佳实践

  • 保证单元测试在沙盒环境中运行
  • 订阅一个observable时，提供成功和失败的callback
  • 当测试组件有service依赖时，使用mock service
  • 访问DOM使用debugElement而不是nativeElement。因为debugElement提供了抽象基础运行环境。
  • 使用By.css而不是querySelector，因为querySelector只在浏览器中有效，当应用在服务上运行时会失败
  • 使用fixture.detectChanges()而不是ComponentFixtureAutoDetect，ComponentFixtureAutoDetect无法检测到同步更新
  • 不要过度使用NO_ERRORS_SCHEMA，它会忽略所有的属性和无法识别的标签

RxJS 弹珠测试

TestScheduler & RunHelpers

TestScheduler有一个名为run的回调函数，接受一个RunHelpers类型的参数，该参数提供了弹珠测试所需的所有辅助方法。

  • cold/hot ：根据给定的弹珠图创建冷/热Observable。接受一个弹珠图作为第一参数，values和errors作为附加可选参数。如果我们创建一个热的Observable，^则可将其用于指示“零帧”。
  • flush：开始一个虚拟时间。仅当在run回调之外使用帮助程序和RxJs测试工具时，或者想flush多次使用时，才需要调用。
  • ExpectObservable：用于断言Observable符合弹珠图。
  • ExpectSubscriptions：用于断言Observable具有预期的订阅。

RxJs 学习

操作符

操作符的分类

  • 按功能分类
    • 创建类：of、from
    • 转化类： map
    • 过滤类：filter
    • 合并类：merge，combineLatest，fork
    • 错误处理类：catchError
    • 多播类：publishReplay？
    • 辅助工具类
    • 条件分支类
    • 数学和合计类
    • 背压控制类：zip
      • 背压：发大水时，下游无法让水流即使通过，水流淤积，逐步把压力传递给上游，这种现象称为背压
    • 可连接类：connnect，全部和多播有关
    • 高阶Observable
      • 指产生数据本身也是Observablele对象的Observable对象。
  • 静态和实例分类：都返回一个 Obsevable对象
    • 静态：只能出现在首部，不需要Observable实例就可以执行的函数，如of
      • Observable.of = functionOf
    • 实例：Obsevable的实例函数，前提是有一个创建好的observable对象，如map
      • Observable.prototype.map = functionMap
    • 这两个形式没有本质区别，在链式调用的中间位置只能用实例函数形式的操作符

操作符的实现

每个操作符都是一个函数，实现一个操作符必须考虑以下几点

  • 返回一个全新的Observable对象
  • 对上游和下游的订阅及退订
    • 当不需要从某一个observable获取数据的时候，就退订这个observable
  • 处理异常情况
    • 捕获异常的错误并将其传递给下游
    • 将上游的异常传递给下游
  • 及时释放资源
    • DOM操作要注意相关Observeable的及时释放
    • Websocket等也是

关联 Observable

把上面的函数与 Observable 关联起来

####打补丁（会污染全局Observable

####bind绑定特定 Observable 对象（无法使用链式调用

  • 缺点：无法使用链式调用（绑定操作符可解决问题，但是还未进入标准）

lift（应用层开发者很少使用

多是RxJs库的开发者使用，访问了this，违背了函数式编程的原则

pipeable 操作符（letable操作符）

前面几个关联Observable 的方法实际上都违背了函数式编程的原则，因为在实现上都访问了this，那就多了一个改变函数行为的因素，不算纯函数了。

操作符和 Observable 关联（打补丁）的缺点
  1. 不能应用treeShaking
  2. 相当于修改了全局资源（Observable）
  3. 使用了 this ，不是纯函数

单线程的 Javascript

JavaScript本身是一门单线程语言。当然，某些浏览器具有web worker功能，可以将没有web worker理解为一个线程，可以把JavaScript中执行的功能分配给多个线程。但是web worker具有一定的局限性：首先并不是所有浏览器都支持web worker，其次web worker访问资源会受限制，比如不能直接操作dom。

多线程的缺点：竞态

当两个异步事件同时发生时，在线程A更新资源X时，线程B被激活，线程B更新完资源X后，线程A重新获得控制权，继续更新资源X，这样可能导致资源X同时被线程A和B更新，最终的数据结果紊乱，失去了数据的一致性。解决的办法利用加锁等机制限制多个线程对同一资源的访问。但是如果某个地方忘记加锁，就可能在关键的时候造成致命的问题。

调用栈和事件循环

JavaScript的解析和运行环境统称为JavaScript引擎。引擎有许多种实现，其中Chrome和Nodejs使用的是V8引擎，edge使用的Chakra（脉轮：人体精神集中点之一）引擎，Firefox使用过gecko（壁虎）、TraceMonkey、JagerMonkey等。每个引擎都必须实现调用栈（call stack）和事件循环（event loop）的概念。

调用栈是每一门编程语言都有的概念。当调用一个函数的时候，就是在调用栈中创造这个函数的运行空间，函数的参数、局部变量的创建都是在调用栈中完成。当一个函数执行完毕，对应的调用栈上该函数的本次运行空间就会被清除。

调用栈的执行方式适合简单的数据运算，当函数之间的调用较多时，栈会变深，当函数执行完毕后，栈会变浅。而面对复杂的逻辑，就要用到事件循环。

在JavaScript中，事件循环的工作方式大致是：事件循环相当于一个死循环，不断从任务队列中取需要处理的任务，然后把这个任务交给调用栈执行，当这个任务结束后，就从任务队列中拿下一个需要处理的任务给调用栈……周而复始，永不停歇

注意，因为JavaScript是单线程的，所以当调用栈中有正在处理的任务时，事件循环不会去拿下一个任务，这就会导致明明setTimeOut的时间已经到了，但是实际上还要等上一个任务结束才能执行，时间并不真正准确。

而任务队列中的任务可以分为：micro task和macro task。事件队列并不只是一条队列，由于引擎的不同，会把不同的任务区分对待，比如setTimeOut和setInterval和时间相关的任务排成一列，网络请求排一队。当调用栈空闲时，事件循环就会从优先级较高的队列中拿下一个任务交给调用栈，当这个队列被清空后，再去下一个优先级较高的队列中拿任务。因此，一个任务想要执行，并不光是等待队列前面的任务，还可能要等待别的队列完成。

上面举例的任务都为macro task。micro task 在概念上只有一条队列，它是优先级最高的，每次调用栈空闲时，事件循环都会优先检查micro task的队列。不同的引擎micro task的定义不同，在V8引擎中，nextTick就是micro task，在某些引擎中 promise的the和catch也是micro task。微任务仅来自我们的代码。

函数式编程

函数式编程的要求
  • 声明式
  • 纯函数
  • 数据不可变性

传统上讲，JS并不是一门纯粹的函数式编程语言，但是函数是JS的一等公民，因为函数是一个对象，可以作为参数被赋值给一个变量，也可以作为参数传递。因此可以应用函数式编程思想中的许多东西。

声明式编程 vs 命令式编程

与声明式编程相对应的编程方式就是命令式编程。

二者的主要区别是，声明式编程更关注What，而命令式更关注How，在命令式编程中可能存在一个很大的问题就是重复的代码，因为它更关注详细的过程。

纯函数

纯函数的主要满足以下两个条件，满足纯函数的特性也叫做引用透明度（referential transparency）

  1. 函数的执行过程完全由入参决定，不会受其他任何数据的影响。
  2. 函数不会修改外部状态

比如JS中的String.split()， 会返回一个新数组，但是不会修改原来的字符串，也不会影响其他变量，这就是个纯函数

而Array.splice()会修改第一个入参，并返回该入参，就不是一个纯函数

与纯函数相对的就是不纯函数，不纯函数包括但不仅限于以下几个特点

  1. 网络输入、输出操作
  2. 操作浏览器的DOM
  3. 读取用户的输入
  4. 抛出一个异常
  5. 改变全局变量的值，改变输入参数引用的对象

其中前四条是比较容易忽略的，实际上判断一个函数纯不纯的方法，就是看讲一个函数调用替换为一个预期返回的常数，看程序的运行结果是否一样（说明不依赖或改变外部参数）

数据的不可变性（Immutable）

当我们需要数据发生改变的时候，保持原有数据不变，产生一个新的数据来体现这种变化。

注意：在JS中const关键字虽然有常数的意思，但其实只是规定一个变量引用的对象不能改变，没有规定这个对象本身不能改变，因此这个常量依然是变量

函数式编程与面向对象编程

面向对象编程是将状态的改变封装起来以达到使代码更加简洁，并复用部分代码的目的。而函数式编程是尽量减少变化的部分，使代码逻辑更清晰。

面向对象的思想是把数据封装在一个类的实例对象中，外部不能直接操作这些数据，只能通过类提供的一些实例方法读取和修改这些数据，这就限制了对数据的访问方式。这样就会有一个缺陷，就是数据的修改历史完全被隐藏了。

当我们在代码中看到一个对象实例的时候，即使知道了对象当前的状态，也没法知道对象是如何一步步走到这状态的，这种不确定性导致代码的可维护性下降。

在函数式白岸城中，倾向于数据与方法分开，函数可以处理数据，但是是作为纯函数产生新的数据，也就是不改变原有的数据（数据的不可变性）

函数响应式编程

优势： 不用关心数据是同步产生的还是异步产生的

使用

深链式导入相应模块
Tree-Shaking

在JS的打包工具中，有一个功能叫Tree-Shaking，指在打包过程中发现没有被引用过的函数，在最终的打包文件中就不会包含这些无用的函数代码。

值得注意的是，Tree-Shaking只对import语句的导入产生作用，对于CommonJS的require函数导入方式不起作用。因为Tree-Shaking的工作方式是对代码进行静态分析，import只能出现在代码第一层，被导入的模块以字符串常量出现，因此满足静态分析的需要。而require还可以出现在条件分支语句中，参数也可以是动态产生的字符串，所以只有在动态执行的时候才知道require函数如何执行，因此Tree-Shaking不会分析到它。

那么RxJS为什么要用深链式导入模块呢，Tree-Shaking为什么对它不起作用呢？

这是因为在RxJs中的大部分函数都是围绕一个Observable类创建的，这些功能都体现为这个类的一个函数，并挂在这个Observable上。也就是说这些函数在内部已经被Observable这个类引用了。

  • 在node_modules/rxjs/Rx.hs中会发现功能模块都是被require进去的
  • 在某一个js文件中（如of），可以发现是先require获得Observable类，然后在Observable上添加一个类函数（如of），那么Observable.of就成为了一个函数，of被引用了，因此最终打包文件会包含of。

因此，使用RxJS需要按需导入，采用deep-link的方式。不过更好的方式是，用一个代码文件专门导入RxJS相关的功能，其他的代码文件再导入这个文件，这样就能把RxJS导入的工作做集中管理。

Observer 和 Observable

RxJS的运行就是Observer和Observable之间的互动游戏

  • Observable：可被观察的对象，在RxJS中就是数据流
  • Observer：观察者
  • subscribe：连接二者的桥梁

Observable实现了两种模式

  • 观察者模式
  • 迭代器模式
观察者模式

将逻辑分为发布者和观察者，发布者只管产生数据，观察者负责处理产生的数据。

Observable负责发布者的责任，产生事件

subscribe的参数负责观察者的责任，响应事件

何时调用subscribe，发布者与观察者之间的关联状态

迭代器模式

迭代器：能够遍历一个数据集合的对象。通常包含

  • getCurrent；获取当前游标所指向的元素
  • moveToNext：将游标移动到下一个元素
  • isDone：是否已经遍历完所有元素

而在RxJS中似乎看不到这样的代码，那是因为RxJS中的数据是推来的，而不是拉。这就是迭代器模式和观察者模式的结合。

null is a object ?

But why there is ‘null is not a obejct’ error?

实际上，这是一个typeof的历史遗留问题，null本身应该是原始数据类型中的一种，而不属于object。

在JavaScript的早期版本中，值以32为存储，其中前3位表示数据类型，其余表示值。对于一个对象，前三位是000，而在JavaScript中，null又表示没有任何内容/无效，因此将其表示为全0，所以才会出现typeof null === 'object'的错误。

InnerHtml InnerText And TextcCntent

在js中，有三种方式设置/得到DOM中HTML元素的内容

InnerHtml

设置/返回一个元素content，以新的HTML形式

HTML5禁止在innerHtml中包含script标签，因为会产生XSS脚本攻击这类安全问题。

innerText

返回内部除了script标签和style标签以外的所有元素的内容，返回的内容是可见的文本，并没有任何的格式化。(注意不会显示HTML代码中多余的空格)

缺点：innerText会触发浏览器的重排（reflow），因为他是css层面的概念，这将会导致性能下降。

reflow指的是浏览器重新计算页面元素及重新呈现文档时会出现的。触发reflow的例子包括调整浏览器的窗口大小或者更改DOM中的元素。这种计算是昂贵的，所以应该将其最小化以提高速度、效率及用户体验。

textContent

textContent将返回所有原始内容，不包括HTML标签。因此返回的值包括空格、换行符、回车符、script和style元素的内容等。与innerHTML不同的是，它具有更好的性能，因为他的值没有被解析为html，因此不会有XSS攻击的问题；与innerText相比，由于它与css无关，所以不会出发reflow。所有的node对象都有textConetnt，但是只有HTMLElement对象有innerText。

关于 setTimeout 在for 循环中的问题

问题1：var 的输出为什么不理想

首先需要两个知识点——事件循环、作用域

  1. setTimeout绑定的回调函数只是往Javascript的事件循环机制中注册了一个定时器，这个定时器只可能在当前时间循环的下一次事件循环中才有可能被执行（需要检查时间是否符合设置的时间点）
  2. js引擎遇到一个异步事件后并不会一直等待其返回结果，而是会将这个事件挂起，继续执行执行栈中的其他任务。当一个异步事件返回结果后，js会将这个事件加入与当前执行栈不同的另一个队列，我们称之为事件队列。被放入事件队列不会立刻执行其回调，而是等待当前执行栈中的所有任务都执行完毕， 主线程处于闲置状态时，主线程会去查找事件队列是否有任务。如果有，那么主线程会从中取出排在第一位的事件，并把这个事件对应的回调放入执行栈中，然后执行其中的同步代码…，如此反复，这样就形成了一个无限的循环。这就是这个过程被称为“事件循环（Event Loop）”的原因。

根据以上两点，for循环执行的过程中，会将setTimeout 注册的5个定时器放入事件队列中，当for循环执行完毕之后，且主线程处于闲置状态时，从事件队列中依次执行setTimeout。因此，当setTimeout函数第一次执行的时候，在执行console.log(i)的时候，这个时候的i其实是全局作用域下的i，这个时候循环是已经结束了，这时候i为5.
输出5次5是因为javacript没有块作用域，只有函数作用域，一个函数在定义时其中变量的作用域就确定；
